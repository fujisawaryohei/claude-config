# Claude Code グローバルアーキテクチャ定義
# 配置先: ~/.claude/rules/architecture.md
# 目的: 全プロジェクト横断で適用する構成要素の責務・設計原則を定義する

---

## 1. 構成要素マップ

```
~/.claude/
├── CLAUDE.md              # 恒久的な行動指針・個人規約
├── rules/
│   └── architecture.md   # このファイル（構成要素の設計原則）
├── skills/                # 全PJ共通の再利用スキル
├── agents/                # 全PJ共通の専用エージェント
└── settings.json          # Hook・権限設定
```

---

## 2. 各構成要素の責務定義

### CLAUDE.md
**責務**: Claudeの恒久的な行動指針・個人スタイルの定義

- セッションをまたいで常に有効な指示を書く場所
- コーディングスタイル・言語・応答スタイルなどの個人規約
- プロジェクト固有の情報は書かない（プロジェクト側の CLAUDE.md に委ねる）
- 長くなるほど指示品質が下がる。「Claudeがすでに正しくやっていること」は書かない

**書くべきもの**
- 使用言語・応答形式の好み
- コーディング規約の原則（詳細は rules/ に分割）
- よく使うコマンド・ツールのリファレンス

**書かないもの**
- lintやテスト実行のルール → Hook に委ねる
- 特定ファイル種別にしか関係しないルール → rules/ に分割する
- プロジェクト固有の情報 → プロジェクト側の CLAUDE.md へ

---

### rules/
**責務**: ファイル種別・ドメイン別に分割した設計規約

- CLAUDE.md の肥大化を防ぐための分割先
- 特定のファイル種別（*.ts, *.py 等）や関心領域にスコープを絞って記述する
- glob 指定で「関連ファイルを触るときだけ読み込まれる」よう設計する

**ファイル構成例**
```
rules/
├── architecture.md   # このファイル（構成要素の設計原則）
├── typescript.md     # TypeScript固有の規約
├── testing.md        # テスト設計の規約
└── security.md       # セキュリティ規約
```

**書くべきもの**
- 特定言語・フレームワークの規約
- アーキテクチャの制約（循環依存禁止等）
- セキュリティ・品質に関する判断基準

**書かないもの**
- 全場面で必要なルール → CLAUDE.md へ
- 「必ず実行させたい」処理 → Hook へ

---

### skills/
**責務**: Claudeが文脈から自律的に呼び出す再利用可能な手順書

- トリガー主体は **Claudeの推論**（description とのマッチング）
- 人間が `/skillname` で明示的に呼ぶこともできる
- 「このタスクが来たら毎回同じ手順を踏む」処理をカプセル化する

**設計原則**
- description はルーティングキー。曖昧な説明はClaudeが素通りする
- description には実際に使われるキーワードを詰め込む
- 1 Skill = 1 責務。汎用スキルは作らない
- 複数 Skill が同じ description キーワードを持つと誤ルーティングが起きる

**description の書き方**
```yaml
# 悪い例（曖昧）
description: コードをきれいにする

# 良い例（トリガー条件とキーワードが明確）
description: |
  TypeScriptファイルを編集した後、コードの重複・未使用変数・
  複雑な条件式を検出して修正する。
  キーワード: simplify, refactor, cleanup, 整理, リファクタ
```

**Commandとの使い分け**
- 自動発火させたい → Skill
- タイミングを自分で制御したい → Command
- 両方必要な場合は Skill で作り、Command から呼び出す

---

### agents/
**責務**: コンテキスト分離・並列実行が必要な専用ワーカー

- トリガー主体は **Claude**（メインコンテキストからの委譲）
- 独立したコンテキストウィンドウ・システムプロンプト・ツール権限を持つ
- メインコンテキストを汚染せずに重いタスクを処理させる

**設計原則**
- 1 Agent = 1 目的。「何でもできるAgent」は作らない
- ツール権限は最小権限で定義する（調査系は Read/Grep/Glob のみ等）
- Skill（context: fork）で代替できるなら Agent は不要
- 並列実行や権限分離に明確な価値があるときだけ作る

**Skillとの使い分け**

| | Skill | Agent |
|---|---|---|
| 何を変えるか | Claudeが知ること | 誰が作業するか |
| コンテキスト | 共有 | 独立 |
| 並列実行 | 限定的 | 可能 |
| 権限制御 | なし | あり |
| 作成コスト | 低 | 高 |

→ 迷ったら Skill から始める。Agent が必要になったら移行する。

---

### hooks/（settings.json 内で定義）
**責務**: イベント駆動で確実に実行させる副作用処理

- トリガー主体は **ライフサイクルイベント**（PostToolUse, PreToolUse 等）
- CLAUDE.md の指示と異なり、**確定的・強制的**に実行される
- 「お願い」ではなく「強制」が必要な処理に使う

**設計原則**
- CLAUDE.md に「必ず〜してください」と書いているものは Hook に移す
- CI/CDで実行される処理と重複させない
- 1 Hook = 1 処理。複数の副作用を1 Hookに詰め込まない

**主要イベントの使い分け**

| イベント | タイミング | 用途例 |
|---|---|---|
| PostToolUse | ファイル編集後 | lint・型チェック自動実行 |
| PreToolUse | ツール実行前 | 危険コマンドのブロック |
| Stop | 応答完了時 | 通知・ログ記録 |
| Notification | 入力待ち時 | デスクトップ通知 |

---

### commands/
**責務**: 人間が明示的にタイミングを制御して起動する処理

- トリガー主体は **人間**（`/commandname` の入力）
- 副作用があり、意図せず自動発火してほしくない処理
- デプロイ・マイグレーション・破壊的操作など

**設計原則**
- 自動発火させたくないものだけここに置く
- Command の中に「Claudeが判断して自動実行してよい」ロジックを書かない
  → それは Skill の責務
- 現在は skills/ に統合されているが、`disable-model-invocation: true` を設定することで
  人間のみが起動できる Command と同等の挙動にできる

**Skillとの使い分け**
- 人間が「今だ」と判断して実行 → Command（または disable-model-invocation な Skill）
- Claudeが「今だ」と判断して実行 → Skill
- 両方ほしい → Skill で作り、`disable-model-invocation` を外す

---

### settings.json
**責務**: Hook定義・ツール権限・モデル設定などの構造化設定

- CLAUDE.md や rules/ に書けない**機械的な設定**を記述する場所
- Hook の定義はすべてここに集約する
- 権限（allowedTools / deniedTools）の管理

**書くべきもの**
- Hook の定義（PostToolUse, PreToolUse 等）
- ツール権限の許可・拒否リスト
- デフォルトモデルの設定

**書かないもの**
- Claudeへの指示・方針 → CLAUDE.md / rules/ へ
- 再利用可能な手順 → skills/ へ

## 3. 追加前の判断フロー

新しい構成要素を追加する前に以下で判断する。

```
追加したい処理がある
│
├─ Claudeへの「方針・規約」として伝えたい
│   ├─ 全場面で必要         → CLAUDE.md
│   └─ 特定領域にスコープ   → rules/
│
├─ 処理を実行させたい
│   ├─ 必ず実行（強制）         → hooks/（settings.json）
│   └─ 条件付きで実行
│       ├─ 人間がタイミング制御  → commands/
│       └─ Claudeが文脈判断     → skills/
│           └─ コンテキスト分離・並列・権限制御が必要
│               → agents/
│
└─ 機械的な設定（権限・モデル等）  → settings.json
```

---

## 4. 禁止パターン

| パターン | 問題 | 正しい対応 |
|---|---|---|
| CommandにClaude自律判断ロジックを混在 | 発火タイミングが非決定的になる | Skillに分離する |
| SkillとCommandで同じ責務を持たせる | どちらが呼ばれるか不明 | どちらかに統一する |
| CLAUDE.mdに「必ず実行してください」 | Hookにすべき処理が混在 | Hookに移す |
| 汎用Agentを作る | 責務が曖昧になる | 目的を1つに絞る |
| CLAUDE.mdに全ルールを詰め込む | 指示品質が均一に下がる | rules/に分割する |
| 複数Skillに同一キーワードを設定 | 誤ルーティングが起きる | キーワードを棲み分ける |

---

## 5. コンテキスト予算の管理

- フロンティアLLMが安定してフォローできる指示数は **150〜200件**
- システムプロンプトがすでに **約50件** を消費している
- CLAUDE.md + rules/ の合計で残り **100〜150件** が上限目安

**予算節約の優先順位**
1. 「Claudeがすでに正しくやっていること」は書かない
2. lintや型チェックはHookに委ねてCLAUDE.mdから削除する
3. 特定領域のルールはrules/に分割してスコープを絞る
4. 長い手順はSkillに切り出してCLAUDE.mdから削除する
